# 互動式專案發想教練

## 線上部署

Vercel Production：<https://personal-brainstorm-coach.vercel.app>

## 啟動方式

### 1. 安裝套件

```bash
npm install
```

### 2. 設定環境變數

在專案根目錄建立 `.env.local`，並填入 OpenAI API Key：

```env
OPENAI_API_KEY=your_openai_api_key
```

### 3. 開發模式啟動

```bash
npm run dev
```

啟動後開啟：

```text
http://localhost:3000
```

### 4. 正式模式啟動

```bash
npm run build
npm run start
```

## Proposal Report

### 動機與目標
<!-- 說明為什麼想做這個專題 -->
本學期需要完成多個專案，但在一開始發想題目時，常常會卡在不知道要做什麼，或是即使有一些模糊想法，也很難進一步整理成具體可執行的專案方向，找了網路上雖然有一些repo在做 project idea generator，但大多只根據少量標籤或簡單偏好產生結果，缺乏更深入的情境式提問與興趣探索，生出來的idea也比較general而且無聊。

因此，本專案希望設計一個「互動式專案發想教練」，透過一系列引導式問題，幫助使用者逐步了解自己的興趣、偏好、能力目標與限制條件，最後生成三個不同風格的專案點子，讓使用者不只是拿到題目，而是能更清楚理解自己為什麼適合這些方向，並進一步開始規劃實作。

此外，也希望可以訓練自己vibe coding的能力，用worktree來管理不同的分支，讓不同的agent在不同的git branch上實作功能，要測試的時候也更方便。

本專案的目標包括：

- 協助使用者從模糊想法中整理出較明確的專案方向。
- 提供比一般題目生成器更具個人化與情境感的推薦結果。
- 讓使用者在獲得專案點子的同時，也能看到初步的技術建議與實作方向。

### 預期功能
<!-- 列出預計實作的功能 -->
- **透過固定問題與情境式提問，蒐集使用者的大致偏好方向**
  - 例如喜歡做工具、遊戲、網站、資料分析或 AI 應用
  - 是否偏好互動感、實用性、視覺效果或技術挑戰
  - 想練習的能力，例如前端、後端、資料結構、演算法或 API 串接
  - Coding的經驗多長，是否有寫過專案
- **根據使用者回答，生成三個不同風格的專案點子**
  - 實用型
  - 創意型
  - 複雜型
- **顯示每個專案點子的簡要說明**
  - 專案核心概念
  - 適合原因
  - 預估難度
  - 為什麼適合
  - 為什麼不適合
- **針對使用者選擇的專案，提供進一步的技術架構建議**
  - 可以使用的技術或框架
  - 可能需要的功能模組
  - 最小可行版本（MVP）建議
- **提供重新生成或調整方向的功能**
  - 使用者若對目前結果不滿意，可以重新選擇偏好或再次生成

### 使用技術

- **前端**

  - TypeScript
  - Tailwind CSS
- **後端**
  - Next.js Route Handlers
- **其他工具**

  1. 前端先蒐集使用者回答
  2. 後端整理成結構化資料後，傳送給 LLM
  3. LLM 回傳三個不同風格的專案提案與技術建議
  4. 前端再將結果以卡片或分區方式呈現給使用者

### Prototype 預計可驗證內容
1. **多步驟問答流程**：前端能順暢展示情境式表單，收集使用者的技術偏好、時間限制與目標。
2. **LLM 動態生成**：後端能將收集到的資料組合成 Prompt，串接 OpenAI API 動態生成專案點子，而非使用固定的假資料。
3. **結果視覺化呈現**：成功解析 LLM 回傳的 JSON 格式，並渲染出包含技術棧、MVP 規劃與難度評估的專案卡片。

### 與課程的關聯

### 競品分析

本專案鎖定「幫助學生從模糊想法→具體可執行專案方向」這個痛點，以下針對三個最具代表性的競品進行分析。

#### 1. ChatGPT（OpenAI）

**定位**：通用型對話式 AI

| 面向 | 說明 |
|------|------|
| 核心功能 | 自由對話，使用者需自己輸入 prompt 才能獲得有意義的結果 |
| 發想引導 | ❌ 無 — 完全依靠使用者主動提問，沒有系統性的引導流程 |
| 個人化程度 | 低 — 不會主動詢問技能程度、偏好或時間限制 |
| 輸出格式 | 非結構化的純文字，需手動整理 |
| 學習曲線 | 需要會「prompt engineering」否則結果非常 general |

**本專案的差異化優勢**：我們預設一套有架構的問答流程，即使完全不懂 prompting 的學生也能獲得個人化、有風格差異的專案建議，而不是一個需要自己問對問題才能用的聊天框。

---

#### 2. Taskade（AI 協作 + 專案管理）

**定位**：連結創意發想到執行的 AI 工作空間

| 面向 | 說明 |
|------|------|
| 核心功能 | AI 生成任務清單、心智圖、自動轉換為專案看板 |
| 發想引導 | ⚠️ 部分有 — 提供結構化模板，但更偏向「已有方向後再規劃」 |
| 個人化程度 | 中 — 模板可設定，但不會根據個人能力做情境式提問 |
| 目標使用者 | 職場團隊 / 已知道要做什麼的人 |
| 免費方案 | 功能受限，核心 AI 功能需付費 |

**本專案的差異化優勢**：Taskade 假設你「已經有 idea 了」，功能在於執行層的協助。我們的專案針對**還沒有想法**或**想法很模糊**的使用者，著重在「發想」本身這個更前期的痛點，尤其針對學生的技能背景與課程限制做客製化推薦。

---

#### 3. GitHub 上的 Project Idea Generator（如 project-idea-generator、IdeaSpark 等開源專案）

**定位**：標籤式或 LLM 加持的專案題目生成工具

| 面向 | 說明 |
|------|------|
| 核心功能 | 根據技術標籤（如 Python、ML、Web）快速生成專案清單 |
| 發想引導 | ❌ 無 — 通常只是給幾個 checkbox 或 dropdown 就生成 |
| 個人化程度 | 低 — 無法感知使用者的動機、學習目標或風格傾向 |
| 推薦品質 | 結果 general、千篇一律，缺乏「適合你的原因」的說明 |
| 技術深度 | 幾乎不提供技術架構建議或 MVP 規劃 |

**本專案的差異化優勢**：這類工具只是「篩選器」而非「教練」。我們透過多輪情境式提問真正理解使用者的偏好與限制，並給出三種**有差異性**的建議（實用型 / 創意型 / 複雜型），同時解釋「為什麼適合你」與「可能的挑戰」——這是純標籤篩選做不到的。

## Prototype Report

### 目前進度
- **前端介面與流程**：完成基於 Next.js 與 Tailwind CSS 的毛玻璃 (Glassmorphism) 質感介面，實作了流暢的多步驟問答表單與狀態管理。
- **後端 API 串接**：完成 `/api/generate-ideas` 路由，整合 Zod 進行資料驗證，並成功串接 OpenAI `gpt-4o-mini` 模型。
- **核心功能打通**：實現從「使用者填寫問答 -> 組裝 Prompt -> LLM 生成 -> 前端解析 JSON 並渲染專案卡片」的完整 End-to-End 流程。

### 遇到的困難
- **確保 LLM 輸出格式的穩定性**：一開始擔心 LLM 回傳的資料結構不固定，會導致前端無法正確渲染卡片（例如陣列缺漏或格式跑掉）。
  - **解決方式**：透過建立嚴格的 Input / Output Schema，並在後端呼叫 OpenAI 時強制開啟 `response_format: { type: "json_object" }`，同時在 System Prompt 內給予極為明確的 JSON 欄位定義，確保 100% 成功解析。

### 下一步計畫
- 優化 Prompt 細節，讓生成的專案點子在「實用型」、「創意型」之間的差異化更明顯。
- 增加「重新生成」或是局部微調的功能，讓使用者能針對單一專案進行更深入的探討。
- 評估是否引入 Firebase 儲存歷史生成紀錄，方便使用者日後回顧。
- 思考當前的提案是否隊使用者有幫助，是否有改進的空間，以及確認自己是用甚麼模型進行生成

## Final Report

### 專案說明

本專案是一個「個人化專題發想教練」，目標是協助學生在期末專題、作品集、黑客松或個人練習前期，快速把模糊的興趣與限制收斂成可實作的專案方向。使用者透過四步驟問卷輸入專案背景、目標、興趣領域、想練習的技能、可用資料來源、成功標準、課程要求與時間限制後，系統會呼叫 OpenAI 模型產生多個量身打造的專題提案。

每個提案會包含專案名稱、核心概念、主要功能、建議技術、資料結構或演算法關聯、MVP 範圍與後續擴充方向，讓使用者不只得到一個題目，而是得到可以直接開始規劃與實作的專案藍圖。除了初次產生提案外，系統也支援針對單一點子做深入調整，例如降低難度、增加創意、加強資料結構與演算法關聯、產生開發計畫或報告大綱。

前端使用 Next.js、React、TypeScript 與 Tailwind CSS 製作互動式問卷、結果卡片、載入狀態、錯誤提示與歷史紀錄介面；後端使用 Next.js Route Handlers 建立 `/api/generate-ideas` 與 `/api/refine-idea`，並透過 Zod 驗證輸入資料。LLM 回傳結果會以 JSON 格式解析，並加入輸出重試與錯誤處理，降低模型回傳格式不穩定造成前端無法渲染的風險。使用者也可以將本次結果儲存在瀏覽器本機歷史紀錄中，方便之後重新載入、比較或繼續修改。

### 使用方式

線上版本已部署於 Vercel：

```text
https://personal-brainstorm-coach.vercel.app
```

本機執行步驟如下：

1. 安裝套件

```bash
npm install
```

2. 建立 `.env.local`，並填入 OpenAI API Key

```env
OPENAI_API_KEY=your_openai_api_key
```

3. 啟動開發伺服器

```bash
npm run dev
```

4. 開啟瀏覽器進入

```text
http://localhost:3000
```

5. 若要建立正式版本並啟動 production server

```bash
npm run build
npm run start
```

實際使用流程：

1. 在首頁點選「開始發想」。
2. 依序填寫專案背景、目標、興趣、技能方向、收斂條件與時間限制。
3. 點選「產生點子」後等待系統生成專案提案。
4. 在結果頁查看每個專案卡片，並可重新生成、修改條件或儲存本次結果。
5. 若想延伸單一提案，可使用卡片上的深入操作，產生更具體的調整建議、開發計畫或報告大綱。
6. 點選「查看歷史紀錄」可以載入或刪除先前儲存在目前瀏覽器的結果。

### 與課程的關聯總結

這個專題和進階程式設計課程的關聯主要在於完整實作了一個可部署、可互動、具備前後端整合的 Web 應用，而不只是靜態頁面。專案中使用 TypeScript 定義資料型別，透過 React state 管理多步驟表單、載入狀態、錯誤狀態、結果列表與歷史紀錄，也使用 API route 將前端輸入轉換成後端可處理的請求流程。這些實作涵蓋了模組化設計、非同步程式、錯誤處理、資料驗證、環境變數管理與部署流程。

和資料結構課程的關聯則體現在資料建模與資料操作上。問卷資料、專案提案、refinement 結果與歷史紀錄都被設計成明確的物件結構；多選欄位使用陣列管理，結果卡片與歷史紀錄也以陣列進行新增、遍歷、載入與刪除。前端使用 key 對應不同專案與其 refinement 結果，概念上接近 map / dictionary 的查找方式；本機歷史紀錄則需要序列化與反序列化資料，並維持資料格式一致。

此外，本專案也把資料結構與演算法的概念放進生成邏輯本身：使用者可以指定「演算法與資料結構」或「資料結構」作為課程要求，系統會要求 LLM 在提案中自然補強相關設計，而不是硬塞不必要的技術名詞。整體而言，這個專題展示了如何把課程中學到的資料表示、流程控制、模組拆分、驗證與錯誤處理，整合成一個能實際解決學生專題發想問題的完整應用。
