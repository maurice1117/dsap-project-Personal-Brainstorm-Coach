# [互動式專案發想教練]

## Proposal Report

### 動機與目標
<!-- 說明為什麼想做這個專題 -->
本學期需要完成多個專案，但在一開始發想題目時，常常會卡在不知道要做什麼，或是即使有一些模糊想法，也很難進一步整理成具體可執行的專案方向，找了網路上雖然有一些repo在做 project idea generator，但大多只根據少量標籤或簡單偏好產生結果，缺乏更深入的情境式提問與興趣探索，生出來的idea也比較general而且無聊。

因此，本專案希望設計一個「互動式專案發想教練」，透過一系列引導式問題，幫助使用者逐步了解自己的興趣、偏好、能力目標與限制條件，最後生成三個不同風格的專案點子，讓使用者不只是拿到題目，而是能更清楚理解自己為什麼適合這些方向，並進一步開始規劃實作。

此外，也希望可以訓練自己vibe coding的能力，用worktree來管理不同的分支，讓不同的agent在不同的git branch上實作功能，要測試的時候也更方便。

本專案的目標包括：

- 協助使用者從模糊想法中整理出較明確的專案方向。
- 提供比一般題目生成器更具個人化與情境感的推薦結果。
- 讓使用者在獲得專案點子的同時，也能看到初步的技術建議與實作方向。

### 預期功能
<!-- 列出預計實作的功能 -->
- **透過固定問題與情境式提問，蒐集使用者的大致偏好方向**
  - 例如喜歡做工具、遊戲、網站、資料分析或 AI 應用
  - 是否偏好互動感、實用性、視覺效果或技術挑戰
  - 想練習的能力，例如前端、後端、資料結構、演算法或 API 串接
  - Coding的經驗多長，是否有寫過專案
- **根據使用者回答，生成三個不同風格的專案點子**
  - 實用型
  - 創意型
  - 複雜型
- **顯示每個專案點子的簡要說明**
  - 專案核心概念
  - 適合原因
  - 預估難度
  - 為什麼適合
  - 為什麼不適合
- **針對使用者選擇的專案，提供進一步的技術架構建議**
  - 可以使用的技術或框架
  - 可能需要的功能模組
  - 最小可行版本（MVP）建議
- **提供重新生成或調整方向的功能**
  - 使用者若對目前結果不滿意，可以重新選擇偏好或再次生成

### 使用技術
<!-- 使用的語言、框架、工具等 -->
- **前端**
  - HTML / CSS / JavaScript
  - React
  - Tailwind CSS
- **後端**
  - Node.js
  - Express
- **其他工具**
  - OpenAI API 或其他大型語言模型 API，用於生成個人化專案點子與技術建議
  - Git / GitHub 進行版本控制
  - Vercel 或 Render 作為部署平台
- **資料處理方式**
  1. 前端先蒐集使用者回答
  2. 後端整理成結構化資料後，傳送給 LLM
  3. LLM 回傳三個不同風格的專案提案與技術建議
  4. 前端再將結果以卡片或分區方式呈現給使用者

### 時程規劃
<!-- 各週預計完成的進度 -->
- **第 1 週：系統設計與前端介面實作**
  - 確定核心功能並設計問答邏輯
  - 利用 Vibe Coding 快速搭建首頁、多步驟問題表單與結果頁面
  - 建立基本版型與前端狀態管理

- **第 2 週：後端建立與 LLM API 串接優化**
  - 建立後端伺服器與串接 LLM API
  - 調整 Prompt 邏輯，優化推薦結果的差異性與實用性
  - 串接前後端，完成使用者輸入到結果生成的完整流程

- **第 3 週：體驗優化、測試部署與成果整理**
  - 完善重新生成與選擇功能，優化整體互動流程
  - 系統整體測試，並快速部署至線上平台
  - 整理專案報告與展示成果

### 與課程的關聯
<!-- 你的專題可能涉及哪些資料結構或演算法概念？為什麼？ -->
- **陣列（Array）**
  - 用於儲存問題列表、使用者回答、生成結果與不同風格的專案提案。
  - 也可用來管理整個問答流程的順序。
- **字典（Hash Map）**
  - 用於記錄使用者的偏好資料，例如興趣、技能方向、時間限制與風格選擇。
  - 也可用於整理每個專案點子的屬性，例如名稱、描述、難度與技術建議。
- **字串處理**
  - 在整理使用者輸入與組合 prompt 時，需要進行字串處理與格式化。
  - 對於 LLM 回傳結果，也需要解析與重整成適合顯示的格式。
