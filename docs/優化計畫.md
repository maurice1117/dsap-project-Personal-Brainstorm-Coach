# 互動式發想品質優化計畫

本文件整理實際試用後的優化方向，目標是讓「互動式專案發想教練」不只是產生三個格式正確的專案點子，而是更像一位能追問、收斂、比較並引導下一步的發想教練。

目前優先採納三個方向：

- 強化問卷，讓系統更了解使用者真正想做與不想做的範圍。
- 強化 Prompt 與輸出規則，避免產生過於通用的題目。
- 針對單一專案點子提供深入追問與後續規劃。

---

## 目前問題

目前系統已能完成「填寫問卷 -> 呼叫 API -> 產生三個專案卡片」的流程，但生成結果仍可能出現以下問題：

- 題目偏通用，例如食譜推薦、預算追蹤、學習路徑生成器，缺乏明確個人化亮點。
- 問卷主要收集興趣、技能與時間，還不夠了解使用者的限制、目標受眾與成功標準。
- 使用者看完三個點子後，若想深入其中一個方向，目前只能重新生成或回表單修改條件。
- 專案卡片雖有 MVP 與技術棧，但還不足以直接引導使用者開始實作或準備課程展示。

---

## Phase 1: 強化問卷收斂能力

**目標**：讓系統在生成前取得更多能影響題目品質的關鍵資訊，減少通用答案。

**實作項目**：

- 在問卷中新增「收斂條件」步驟。
- 新增使用者不想做的類型或技術，例如純 CRUD、登入會員、重度 ML、複雜部署、付費 API。
- 新增目標使用者，例如自己、同學、老師、社團、一般使用者。
- 新增可用資料來源，例如手動輸入、公開 API、CSV、課程資料、使用者自行建立資料。
- 新增成功標準，例如拿高分、可放作品集、可現場 demo、練特定技術。
- 新增偏好成果形式，例如網站、互動視覺化、Dashboard、小工具、遊戲化體驗。
- 新增課程要求或必須展示的概念，例如資料結構、演算法、資料庫、API、前端互動。
- 將新增欄位送到 `/api/generate-ideas`，並加入 Prompt 中讓 LLM 使用。

**驗證方式**：

- 填寫新增欄位後送出，確認 API 接收到完整資料。
- 使用同一組目標輸入時，生成結果應更明確避開使用者不想做的方向。
- 檢查 Prompt 內容有使用目標使用者、成功標準與課程要求。

**預期成果**：

- 生成結果更貼近使用者場景。
- 使用者能更清楚表達「不要什麼」，降低產生不適合點子的機率。
- 後續 Phase 2 可以直接使用這些欄位強化輸出 schema。

---

## Phase 2: 升級 Prompt 與 Output Schema

**目標**：讓每個專案提案不只格式完整，也要具備明確差異化、課程關聯與下一步行動。

**實作項目**：

- 修改 Prompt，要求每個提案至少引用兩個使用者輸入條件。
- 明確禁止泛用題目，除非提案能說明差異化設計。
- 要求三個提案在使用情境、技術亮點、資料結構或演算法上有明顯差異。
- 避免 MVP 預設登入或註冊，除非題目真的需要。
- 新增輸出欄位：
  - `why_this_is_not_generic`
  - `core_algorithm_or_data_structure`
  - `first_3_steps`
  - `week_plan`
  - `scope_down_plan`
  - `demo_highlight`
- 同步更新 Zod schema、TypeScript type 與 ProjectCard 顯示內容。

**驗證方式**：

- 使用測試輸入確認三個專案不再只是一般常見題目。
- 檢查每張卡片都有具體的資料結構或演算法關聯。
- 檢查 `week_plan` 與 `first_3_steps` 能直接作為開發起點。
- 跑 `npm run test` 與 `npm run build`。

**預期成果**：

- 使用者拿到的不只是點子，而是可以開始執行的初步專案計畫。
- 生成結果更適合作為課程專題報告與作品集展示素材。

---

## Phase 3: 新增單一點子深入發想

**目標**：讓使用者能針對喜歡的專案繼續延伸，而不是只能整批重新生成。

**實作項目**：

- 在每張 ProjectCard 新增深入操作按鈕。
- 支援以下操作：
  - 深入這個點子
  - 降低難度
  - 讓它更有創意
  - 加強資料結構 / 演算法關聯
  - 產生開發計畫
  - 產生報告大綱
- 新增後端 API，例如 `/api/refine-idea`，接收原始表單資料、單一 project 與 refine action。
- 回傳單一深入結果，不影響其他卡片。
- 前端顯示 loading、錯誤提示與深入結果區塊。

**驗證方式**：

- 點擊單一卡片的深入操作後，只更新該卡片的延伸內容。
- 不同操作應產生不同方向的回應。
- API 失敗時不清空原本三個提案。

**預期成果**：

- 使用者能從「挑一個點子」自然前進到「把這個點子變成可做計畫」。
- 產品定位從 idea generator 更接近 project coach。

---

## Phase 4: 整合測試與品質驗證

**目標**：確保新增問卷、Prompt 與深入發想功能不破壞既有流程。

**實作項目**：

- 更新測試資料，涵蓋新增問卷欄位與輸出欄位。
- 增加至少一組高限制條件測試，例如「不要登入、不要重度 ML、必須展示資料結構」。
- 檢查手機與桌面版結果卡片排版。
- 檢查錯誤情境：API timeout、JSON schema 不符合、深入發想失敗。

**驗證方式**：

- 跑 `npm run test`。
- 跑 `npm run build`。
- 使用瀏覽器實際走完完整流程。
- 用至少兩組不同使用者情境手動比較生成品質。

**預期成果**：

- 新增功能穩定可用。
- 生成結果更具個人化、可執行性與課程展示價值。

## 建議 Commit 切分

依照 `docs/COMMIT_RULES.md`，建議拆成以下 commit：

1. `docs: 新增發想品質優化計畫`
   - 新增 `docs/優化計畫.md`
   - 定義 Phase 1 到 Phase 4 的開發順序與驗證方式

2. `flow: 新增問卷收斂條件`
   - 新增問卷欄位
   - 更新前端表單 state
   - 將新增資料送到 API

3. `prompt: 加入收斂條件到生成提示`
   - 更新 API input schema
   - 更新 prompt builder
   - 讓 LLM 使用新增欄位生成更個人化的點子

4. `prompt: 強化專案輸出 schema`
   - 新增下一步、展示亮點與課程關聯欄位
   - 更新 Zod schema 與卡片顯示

5. `feat: 新增單一點子深入發想`
   - 新增 refine idea API
   - 新增卡片操作按鈕
   - 顯示單一點子的深入結果
